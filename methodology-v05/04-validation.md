## 4.1 Три уровня валидации

Валидация в блочном кодинге работает на трёх уровнях. Каждый уровень отвечает на свой вопрос.

### Уровень 1: Блочная валидация

```
Вопрос: блок выполняет свой контракт?

Что проверяется:
  - Вход X → выход Y (соответствие контракту)
  - Граничные случаи (невалидный вход, пустой вход, перегрузка)
  - Метрики (время отклика, потребление ресурсов)
  - Инварианты (что НИКОГДА не должно нарушаться)

Кто проверяет: ревьюер-агент
Когда: после каждой реализации или изменения блока
Результат: блок ✅ готов / ❌ возврат кодеру с диагностикой
```

### Уровень 2: Стыковая валидация

```
Вопрос: блоки совместимы между собой?

Что проверяется:
  - Формат: выход A совместим со входом B
  - Транспорт: A отдаёт по тому протоколу, который B ожидает
  - Версии: B работает с актуальной версией контракта A
  - Семантика: данные из A имеют смысл для B (не только формат, но значения)

Кто проверяет: оркестратор
Когда: при создании блока, при breaking change, при сборке
Результат: стык ✅ совместим / ❌ нужен адаптер или обновление

Подробно: 02-decomposition.md, раздел 2.5
```

### Уровень 3: Системная валидация

```
Вопрос: система достигает GOAL?

Что проверяется:
  - Все блоки ✅ → но работает ли система целиком?
  - Измеримые критерии из GOAL.md выполнены?
  - End-to-end сценарии: пользователь делает X → получает Y

Кто проверяет: человек-архитектор (с помощью оркестратора)
Когда: перед релизом, при ретроспективе
Результат: система ✅ достигает цели / ❌ нужна корректировка

Важно: системная валидация — не сумма блочных.
Все блоки могут работать по контрактам, но система не достигает цели.
Пример: каждый блок отвечает за 200ms, но цепочка из 10 блоков 
        даёт 2 секунды — пользователь уходит.
```

### Иерархия проверки

```
GOAL        → системная валидация (правильную ли систему построили?)
  ↓
BLOCKS      → стыковая валидация (все ли части совместимы?)
  ↓
CONTRACT    → блочная валидация (каждый блок выполняет обещание?)
```

Каждый уровень проверяется предыдущим. Если GOAL изменился — пересматриваются блоки. Если блок изменился — пересматриваются стыки.

## 4.2 Микросимуляции

Микросимуляция — проверка поведения блока через эмуляцию входа и проверку выхода, без полной интеграции в систему.

### Зачем

```
Проблема: полная интеграция для проверки одного блока — дорого.
Нужно поднять зависимости, настроить окружение, прогнать весь пайплайн.

Решение: симулировать стыки. Подать на вход блока тестовые данные,
проверить что выход соответствует контракту. Без реальных зависимостей.
```

### Что это

```
Микросимуляция = набор пар (вход → ожидаемый выход),
определённых в контракте блока.

Это НЕ unit-тесты (проверяют код).
Это НЕ интеграционные тесты (требуют всю систему).
Это контрактные тесты — проверяют ПОВЕДЕНИЕ блока как чёрного ящика.
```

### Формат в контракте

```markdown
## Микросимуляции

### Основной сценарий
- Вход: {"username": "alice", "password": "correct"}
- Ожидаемый выход: {"token": "<JWT>", "expires_in": 3600}
- Проверка: status 200, token — валидный JWT, expires_in > 0

### Невалидный вход
- Вход: {"username": "alice", "password": "wrong"}
- Ожидаемый выход: {"error": "invalid_credentials"}
- Проверка: status 401, время отклика < 100ms

### Граничный случай
- Вход: {"username": "", "password": ""}
- Ожидаемый выход: {"error": "validation_error"}
- Проверка: status 400, НЕ раскрывает внутреннюю информацию

### Нагрузка
- Вход: 100 одновременных запросов с валидными данными
- Ожидаемый выход: все 100 получают корректный ответ
- Проверка: 95-й перцентиль < 200ms, 0 ошибок
```

### Кто пишет микросимуляции

```
Оркестратор — при создании контракта.
Микросимуляции — часть контракта, не часть реализации.

Почему оркестратор, а не агент:
  - Агент заинтересован в том, чтобы тесты проходили
  - Оркестратор заинтересован в том, чтобы тесты были честными
  - Разделение ролей: кто пишет код ≠ кто определяет проверки
```

### Кто выполняет микросимуляции

```
Ревьюер-агент — после получения реализации от кодера.

Процесс:
  1. Кодер реализует блок
  2. Ревьюер берёт микросимуляции из контракта
  3. Ревьюер прогоняет каждую симуляцию
  4. Результат: ✅ все прошли / ❌ список провалов с диагностикой
```

### Когда выполняются

```
1. После первой реализации блока — обязательно
2. После каждого изменения блока — обязательно
3. При breaking change зависимости — проверить что блок ещё работает
4. При подозрении на регрессию — прогнать по запросу оркестратора
```

## 4.3 Метрики качества

Когда внутри блока бинарник — код-ревью невозможно. Когда внутри блока код, сгенерированный AI — код-ревью неэффективно (AI генерирует рабочий, но нечитаемый код). Метрики заменяют ревью как инструмент контроля качества.

### Принцип из Манифеста

> Контроль качества — через метрики. Скорость отклика, потребление ресурсов, корректность выхода, устойчивость к граничным случаям. Метрики объективны и не зависят от формы реализации.

### Четыре группы метрик

#### Корректность (блок делает то, что обещает)

```
- Соответствие контракту: вход X → выход Y
- Инварианты не нарушены
- Граничные случаи обработаны
- Ошибки возвращают описательные сообщения

Измеряется: микросимуляции (раздел 4.2)
Порог: 100% обязательных сценариев пройдено
       (нет компромисса — контракт либо выполняется, либо нет)
```

#### Производительность (блок работает достаточно быстро)

```
- Время отклика (p50, p95, p99)
- Пропускная способность (запросов в секунду)
- Время холодного старта (для бинарных блоков)

Измеряется: нагрузочные симуляции
Порог: определяется в контракте (секция "Ограничения")
       Если SLA нет — порог не проверяется
```

#### Ресурсоёмкость (блок не жрёт больше положенного)

```
- Потребление RAM (пиковое и среднее)
- Использование CPU
- Размер бинарника / контейнера
- Сетевой трафик (если релевантно)

Измеряется: мониторинг при нагрузочных симуляциях
Порог: определяется контекстом деплоя
       VPS за $5 — одни лимиты
       Kubernetes кластер — другие
```

#### Устойчивость (блок не падает от неожиданного)

```
- Невалидный вход → описательная ошибка (не crash)
- Недоступная зависимость → graceful degradation или ошибка
- Параллельный доступ → корректная обработка (если релевантно)
- Утечки памяти при длительной работе (для долгоживущих блоков)

Измеряется: симуляции граничных случаев + chaos-тестирование (Tier 3)
Порог: блок НИКОГДА не крашится от невалидного входа
       (это абсолютное требование, не опциональное)
```

### Метрики vs ревью: когда что

```
Метрики работают ВСЕГДА:
  ✅ Исходный код (Python, Go, TypeScript)
  ✅ Скомпилированный бинарник
  ✅ Вызов внешнего API
  ✅ Нейросетевая модель

Ревью кода работает ТОЛЬКО:
  ✓ Когда есть читаемый исходный код
  ✗ Бинарник — нечего ревьюить
  ✗ AI-код — ревью стиля бессмысленно
  
Вывод: метрики — первичный инструмент.
       Ревью кода — опциональное дополнение, не замена.
```

## 4.4 Карта ситуаций

Типичные ситуации при валидации и алгоритм действий для каждой.

### Ситуация 1: Блок не проходит микросимуляции

```
Симптом: ревьюер прогнал симуляции → часть провалилась

Действия:
  1. Ревьюер формирует диагностику:
     - Какие сценарии провалились
     - Что ожидалось vs что получено
     - Гипотеза причины (если есть)
  2. Кодер получает диагностику + контракт → исправляет
  3. Ревьюер повторяет симуляции
  4. Если после 2 итераций не исправлено → эскалация (раздел 4.5)

Ключевое: ревьюер НЕ правит код. Ревьюер описывает проблему.
Кодер НЕ видит ожидания ревьюера. Кодер видит контракт + диагностику.
```

### Ситуация 2: Стык не совпадает

```
Симптом: оркестратор проверяет стык A→B, формат или транспорт не совпадает

Развилка:
  a) Несовпадение формата (A отдаёт XML, B ждёт JSON)
     → Адаптер-блок между ними (см. 02-decomposition, раздел 2.2)
     
  b) Несовпадение транспорта (A пишет в stdout, B слушает HTTP)
     → Адаптер-блок с конвертацией транспорта
     
  c) Несовпадение версий (B работает с A v1, а A уже v2)
     → Каскадное обновление (см. 02-decomposition, раздел 2.4)
     
  d) Семантическое несовпадение (формат совпал, но данные бессмысленны)
     → Пересмотр контракта одного из блоков
     → Требует решения человека

Правило: если можно решить адаптером — решай адаптером.
Переделка блока — крайняя мера.
```

### Ситуация 3: Каскадный фейл

```
Симптом: breaking change блока A ломает блоки B, C, D

Действия:
  1. Оркестратор определяет масштаб (сколько блоков затронуто)
  2. Оркестратор предлагает варианты:
     a) Обновить все зависимые блоки (полное обновление)
     b) Адаптер v1→v2 (временное решение, не трогаем зависимые)
     c) Откатить изменение A (если ещё можно)
  3. Человек выбирает стратегию, учитывая:
     - Сроки (дедлайн через день → адаптер)
     - Технический долг (адаптер = долг, обновление = чисто)
     - Приоритеты (блок B критичен, C и D — нет)
  4. Оркестратор исполняет выбранную стратегию

Антипаттерн: обновлять всё сразу без приоритизации.
Правильно: критичные блоки сначала, остальные — в бэклог.
```

### Ситуация 4: Системная валидация провалена

```
Симптом: все блоки ✅, все стыки ✅, но end-to-end сценарий не работает

Возможные причины:
  a) Эмерджентная проблема — поведение системы > суммы блоков
     (пример: каждый блок добавляет 100ms → цепочка = 1 секунда)
     → Оптимизация цепочки или пересмотр архитектуры
     
  b) Пропущенный блок — в системе не хватает компонента
     → Добавить блок, обновить BLOCKS.md
     
  c) GOAL некорректен — критерии не отражают реальную потребность
     → Пересмотр GOAL.md (решение человека)
     
  d) Интеграционный баг — блоки работают по контракту,
     но реальные данные отличаются от тестовых
     → Обновить микросимуляции реальными данными

Действие: оркестратор диагностирует → предлагает варианты → человек решает
```

### Ситуация 5: Фоллбэк не помог

```
Симптом: кодер не справился за 2 итерации, фоллбэк (мощная модель) 
тоже не справился

Действия:
  1. Оркестратор собирает всю диагностику:
     - Контракт блока
     - Что именно не получается
     - Что пробовали (кодер, фоллбэк)
     - Почему не работает (гипотеза)
  2. Эскалация на человека с полным контекстом
  3. Человек принимает одно из решений:
     a) Упростить контракт (задача слишком сложная для блока)
     b) Разбить блок на части (декомпозиция недостаточна)
     c) Сменить подход (другой транспорт, другая архитектура)
     d) Реализовать вручную (крайний случай)

Ключевое: эскалация — не провал. Это нормальная часть процесса.
~5% блоков требуют человеческого вмешательства.
```

---

### Ситуация 6: Агент ломает verified-компонент

```
Симптом: после реализации блока обнаружено, что ранее
проверенный и работающий компонент сломан

Причины:
  a) Оператор не знал о verified-статусе компонента
     → CLAUDE.md не содержал факт, или оператор не читал
  b) §4 контракта не содержал запрет на изменение файла
     → Контракт неполный
  c) Оператор решил "улучшить" verified-код
     → Нарушение процесса

Действия:
  1. Откатить изменение (git revert)
  2. Добавить факт в CLAUDE.md секцию Verified (если отсутствовал)
  3. Добавить файл в §4 следующего контракта
  4. Если повторяется — добавить pre-commit hook

→ Подробно: patterns/verified-facts.md
```

## 4.5 Эскалация

Четыре уровня, каждый с чёткими критериями перехода.

### Уровень 1: Кодер

```
Кто: дешёвая модель (DeepSeek / аналог)
Задача: реализовать блок по контракту
Лимит: 1 попытка + 1 исправление по диагностике ревьюера = 2 итерации
Переход на уровень 2: ревьюер дважды отклонил реализацию
```

### Уровень 2: Ревьюер-диагност

```
Кто: средняя модель (Sonnet / аналог)
Задача: 
  - Понять ПОЧЕМУ кодер не справился
  - Сформулировать более точную диагностику
  - Попробовать направить кодера конкретнее
Лимит: 1 попытка диагностики + повтор кодера
Переход на уровень 3: после диагностики кодер всё ещё не справился
```

### Уровень 3: Фоллбэк

```
Кто: мощная модель (Opus / аналог)
Задача:
  - Получает контракт + всю историю попыток
  - Реализует блок самостоятельно
  - Или предлагает изменение контракта (если задача некорректна)
Лимит: 1 попытка
Переход на уровень 4: фоллбэк не справился или предлагает менять контракт

Частота: ~5% блоков доходит до этого уровня
```

### Уровень 4: Человек

```
Кто: архитектор
Задача:
  - Получает полный контекст от оркестратора
  - Принимает архитектурное решение
  - Варианты: упростить контракт, разбить блок, сменить подход, 
    реализовать вручную

Важно: человек получает не "помоги написать код",
а "вот проблема, вот что пробовали, вот варианты решения — выбери".
Оркестратор делает анализ, человек принимает решение.
```

### Таймауты

```
Кодер:          до 15 минут на итерацию (для типичного блока)
Ревьюер:        до 5 минут на диагностику
Фоллбэк:       до 30 минут на реализацию
Человек:        без таймаута (это бизнес-решение)

Таймауты — ориентир, не жёсткое правило.
Сложный блок может требовать больше времени.
Если кодер работает 2 часа — что-то не так с контрактом, не с кодером.
```

### Экономика эскалации

```
Зачем дешёвая модель первой:
  - 80% блоков реализуются с первой попытки дешёвой моделью
  - Стоимость: ~$0.01 за блок
  
  15% блоков требуют 2 итерации + ревьюер:
  - Стоимость: ~$0.05 за блок
  
  4% блоков требуют фоллбэк:
  - Стоимость: ~$0.50 за блок
  
  1% блоков требуют человека:
  - Стоимость: время архитектора

Средняя стоимость: ~$0.03 за блок
При 15 блоках проекта: ~$0.45 за весь проект
  (не считая стоимости оркестратора)
```

## 4.6 Исполняемые контракты

### Проблема

Контракты сейчас — markdown-файлы. Человек или AI читает их и интерпретирует. Это работает, но создаёт зазор: контракт описывает одно, реализация делает другое, и расхождение обнаруживается поздно.

### Спектр исполняемости

```
Уровень 0 (сейчас): Бумажный контракт
  CONTRACT.md — текстовое описание
  Валидация: ревьюер читает контракт, проверяет вручную
  Работает: да
  Проблема: зависит от качества интерпретации AI

Уровень 1: Контракт + JSON Schema
  CONTRACT.md + schema.json
  Валидация: автоматическая проверка формата входа/выхода
  Что даёт: 
    - Вход не соответствует схеме → ошибка ДО выполнения
    - Выход не соответствует схеме → ошибка ПОСЛЕ выполнения
  Не покрывает: семантику, производительность, инварианты

Уровень 2: Контракт + Schema + автотесты
  CONTRACT.md + schema.json + tests/
  Микросимуляции из контракта → автоматические тесты
  Валидация: прогнать тесты = проверить контракт
  Что даёт: корректность + граничные случаи автоматически
  Не покрывает: производительность, ресурсы (нужен бенчмарк)

Уровень 3: Полностью исполняемый контракт
  CONTRACT.md + schema.json + tests/ + benchmarks/ + monitors/
  Валидация: CI/CD пайплайн, мониторинг в продакшене
  Что даёт: полная автоматическая проверка всех метрик
  Когда нужен: Tier 3 (40+ блоков), критичные системы
```

### Путь миграции

```
Tier 1 (лендинги): Уровень 0 достаточен
  - 3-5 блоков, ручная проверка справляется
  - Overhead от автоматизации > пользы

Tier 2 (типичный проект): Уровень 1-2
  - 8-15 блоков, ручная проверка начинает тормозить
  - JSON Schema для входов/выходов — низкий порог входа
  - Автотесты из микросимуляций — окупаются за 2-3 итерации

Tier 3 (enterprise): Уровень 2-3
  - 40+ блоков, ручная проверка невозможна
  - CI/CD обязателен
  - Мониторинг метрик в продакшене

Принцип: не автоматизируй, пока боль не заставит.
Лучше бумажный контракт, чем автоматизация ради автоматизации.
```

### Генерация тестов из контракта

```
Микросимуляции в контракте уже содержат всё нужное:
  - Вход (тестовые данные)
  - Ожидаемый выход (эталон)
  - Проверки (что именно сравнивать)

Шаг к автоматизации:
  1. Оркестратор читает микросимуляции из CONTRACT.md
  2. Генерирует тестовый файл (Python pytest / Go test / bash)
  3. Тест запускается как обычный автотест
  4. Результат: контракт проверяется автоматически

Контракт остаётся в markdown (читаемый человеком).
Тесты генерируются автоматически (исполняемый машиной).
Одни и те же данные — два формата.
```

## 4.7 Валидация по тирам

### Tier 1 (1-5 блоков, лендинг)

```
Блочная валидация:
  - Ручная: запустил → проверил → работает
  - Микросимуляции: в голове или в заметках, не формализованы
  
Стыковая валидация:
  - Ручная: 3-5 стыков можно проверить глазами
  - Оркестратор проверяет при сборке
  
Системная валидация:
  - Открыл в браузере → выглядит правильно → работает
  - PageSpeed, мобилка, формы — ручной чеклист

Эскалация:
  - Кодер → сразу фоллбэк (при 3-5 блоках нет смысла 
    в промежуточных уровнях)

Инструменты: ничего специального, ручная проверка
```

### Tier 2 (8-15 блоков, типичный проект)

```
Блочная валидация:
  - Микросимуляции обязательны в каждом контракте
  - Ревьюер-агент прогоняет симуляции
  - JSON Schema для входов/выходов (Уровень 1 исполняемости)
  
Стыковая валидация:
  - Оркестратор проверяет ВСЕ стыки после каждого breaking change
  - Автоматический граф зависимостей (из BLOCKS.md)
  - Адаптеры создаются как отдельные блоки
  
Системная валидация:
  - End-to-end сценарии (3-5 основных путей пользователя)
  - Метрики производительности цепочки (не только отдельных блоков)
  - Чеклист перед деплоем

Эскалация:
  - Полная цепочка: кодер → ревьюер → фоллбэк → человек
  - Лимиты итераций соблюдаются

Инструменты: 
  - Микросимуляции в контрактах
  - JSON Schema (опционально, рекомендуется)
  - RAG для поиска прошлых проблем
```

### Tier 3 (40+ блоков, enterprise)

```
Блочная валидация:
  - Автотесты генерируются из контрактов (Уровень 2-3 исполняемости)
  - CI/CD прогоняет тесты при каждом коммите
  - Мониторинг метрик в продакшене (алерты при деградации)
  
Стыковая валидация:
  - Автоматическая проверка совместимости при merge
  - Контрактные тесты между командами
  - Schema registry (реестр всех контрактов с версиями)
  
Системная валидация:
  - Автоматические end-to-end тесты
  - Canary deploys (новая версия блока на 5% трафика)
  - Chaos engineering (намеренное отключение блоков)
  
Эскалация:
  - Автоматическая маршрутизация задач между агентами
  - Метрики эскалации: % блоков на каждом уровне
  - Если >10% доходит до фоллбэка — проблема в контрактах, не в кодере

Инструменты:
  - CI/CD pipeline
  - Schema registry
  - Мониторинг + алерты
  - Распределённый RAG (по командам / подсистемам)
```

## 4.9 Verified Facts — защита проверенного

Три уровня валидации (4.1) проверяют корректность НОВОЙ работы. Но есть комплементарная задача: защитить СТАРУЮ проверенную работу от разрушения новой.

Verified fact — факт, подтверждённый ручной проверкой, который имеет приоритет над любым автоматическим решением. "PUDU HMAC: path WITH query params" — это не мнение, а факт, выверенный методом проб и ошибок.

Три механизма защиты:

```
1. Секция Verified в CHANGELOG (append-only)
   — полный список всех проверенных фактов

2. Секция Verified в CLAUDE.md
   — критичные факты, которые агент видит каждую сессию

3. §4 операционного контракта
   — файлы/компоненты с verified-статусом в списке "НЕ трогать"
```

Verified facts дополняют валидацию: валидация проверяет "новое работает", verified facts гарантируют "старое не сломано".

→ Подробно: `patterns/verified-facts.md`

## 4.10 Антипаттерны валидации

### ❌ "Тесты проверяют реализацию, а не поведение"

```
Проблема: тест проверяет что функция вызвана 3 раза, 
          а не что выход корректен
Результат: при смене реализации (Python → бинарник) все тесты ломаются,
           хотя поведение не изменилось
Правило: тесты проверяют ВХОД → ВЫХОД. 
         Никогда — внутреннее устройство блока.
```

### ❌ "Валидация после интеграции, а не до"

```
Проблема: собрали систему, потом начали тестировать
Результат: баг на стыке блоков 2 и 5 обнаруживается, 
           когда уже всё собрано
Правило: валидация каждого блока ДО интеграции.
         Валидация стыков ДО сборки.
         Системная валидация — финальная проверка, не первая.
```

### ❌ "100% покрытие микросимуляциями"

```
Проблема: попытка покрыть ВСЕ возможные входы
Результат: 200 симуляций, из которых 180 тривиальны
Правило: покрывать основной сценарий + граничные случаи + 
         ошибочные входы. 5-15 симуляций на блок достаточно.
         Больше — если контракт сложный.
```

### ❌ "Фоллбэк как норма"

```
Проблема: >10% блоков доходит до фоллбэка
Результат: дорого, медленно, система не масштабируется
Диагностика: проблема в контрактах, не в кодерах.
  - Контракты слишком сложные → упростить, разбить блок
  - Контракты неоднозначные → переписать чётче
  - Контракты содержат противоречия → исправить
Правило: если фоллбэк частый — чини контракты, не увеличивай бюджет.
```

### ❌ "Валидация = бюрократия"

```
Проблема: "мы маленькая команда, нам не нужна формальная валидация"
Результат: при 10 блоках — ещё терпимо.
           При 15 — начинаются неуловимые баги на стыках.
           При 40 — хаос.
Правило: масштаб валидации растёт с тиром.
         Tier 1 — неформально. Tier 2 — микросимуляции обязательны.
         Tier 3 — полная автоматизация.
         Но КАКАЯ-ТО валидация нужна ВСЕГДА.
```
