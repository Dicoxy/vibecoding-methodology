## 2.1 Определение блока

Блок — логическая единица системы с одной ответственностью и чётким контрактом.

**Определение исключительно логическое.** Нет лимитов по строкам, токенам или размеру файла. Причина: при бинарном содержимом блока (скомпилированный бинарник) токены и строки не имеют смысла как единица измерения. Блок определяется тем, что он делает, а не тем, сколько в нём кода.

### Что определяет блок

```
Блок — это:
  - Одна логическая ответственность (одна причина для изменения)
  - Контракт: вход, выход, транспорт
  - Цель: зачем этот блок существует (наследуется от GOAL.md)
  - Чёрный ящик: снаружи виден только контракт, не реализация

Внутри блока может быть:
  - Код на любом языке (Python, TypeScript, Rust, ...)
  - Скомпилированный бинарник
  - Вызов внешнего API
  - Нейросетевая модель
  - Комбинация вышеперечисленного
```

### Размер как сигнал

Размер блока — не правило, а диагностический сигнал.

```
Блок "чувствуется большим" →
  Задай вопрос: что здесь содержание, что обёртка,
  что входящие/исходящие данные?
  
  Если находишь ответ → декомпозируй дальше
  Если блок неделим без потери целостности → оставь как есть
```

Нет формулы "блок должен быть X строк". Есть принцип: **если блок можно разделить на две части с разными ответственностями — его нужно разделить.**

## 2.2 Контракт блока

Контракт — обещание блока внешнему миру. Фиксирует не только формат данных, но и транспорт.

### Структура контракта

```markdown
# CONTRACT: [имя блока]
Версия: v[N]

## Цель
[Зачем этот блок существует — наследуется от GOAL.md проекта]

## Вход
- Формат: [JSON schema / protobuf / binary]
- Транспорт: [HTTP POST /endpoint | stdin | pipe | function call]
- Обязательные поля: [...]
- Опциональные поля: [...]

## Выход
- Формат: [JSON schema / protobuf / binary]
- Транспорт: [HTTP 200 | stdout | pipe | return value]
- Гарантии: [что всегда возвращается]
- Инварианты: [что никогда не нарушается]

## Ограничения
- [SLA, если есть: время отклика, пропускная способность]
- [Ресурсы: память, CPU, если критично]

## Зависимости
- [От каких контрактов зависит: contract-name v[N]]

## Не делает
- [Явные границы ответственности]
```

### Транспорт как часть контракта

Коннектор — не отдельная сущность. Транспорт — свойство контракта.

```
Почему транспорт в контракте:
  - Оркестратор должен знать КАК вызвать блок, не только ЧТО передать
  - При смене runtime (Python → бинарник) транспорт может измениться
  - Валидация стыков проверяет совпадение И формата, И транспорта

Типичные транспорты:
  HTTP + JSON       — универсальный, обязателен на стыке с фронтендом
  stdin/stdout      — для CLI-блоков и бинарников
  function call     — для блоков в одном процессе
  pipe / socket     — для быстрого общения между бинарниками
  message queue     — для асинхронных блоков (Tier 3+)
```


### Два уровня контрактов

Контракт выше — **архитектурный**: описывает интерфейс блока в системе. Его аудитория — оркестратор. Его задача — валидация стыков, версионирование, каскадные обновления.

Когда блок передаётся AI-оператору для реализации, архитектурного контракта недостаточно. Оператору нужен **операционный контракт**: что уже готово (с реальными JSON), что конкретно сделать (пошагово с кодом), что НЕ трогать (явные запреты), как проверить результат (тест приёмки с конкретными данными).

```
Архитектурный контракт (этот раздел):
  Для кого: оркестратор
  Содержит: вход/выход/транспорт/ограничения
  Когда: при проектировании системы

Операционный контракт (→ patterns/contract-spec.md):
  Для кого: AI-оператор (Claude Code, Cline, человек + AI)
  Содержит: §1 что готово, §2 маппинг, §3 что делать,
            §4 что НЕ делать, §5 тест приёмки, §6 справка
  Когда: при передаче блока на реализацию
```

Оба уровня дополняют друг друга. Операционный контракт ссылается на архитектурный, но добавляет конкретику для исполнителя.

→ Подробно: `patterns/contract-spec.md`

### Адаптер-блок

Когда выход блока A не совпадает со входом блока B (по формату или транспорту) — между ними вставляется адаптер-блок.

```
Адаптер — это обычный блок:
  - Имеет свой контракт (вход = выход A, выход = вход B)
  - Имеет свою реализацию
  - Может быть заменён
  - Не является специальным типом

Кто создаёт:
  - Оркестратор обнаруживает несовпадение при валидации стыков
  - Оркестратор генерирует контракт адаптера
  - Агент реализует адаптер как любой другой блок
```

## 2.3 Версионирование контрактов

Только major-версии. Привязано к changelog.

### Правила

```
1. Версия меняется ТОЛЬКО при breaking change
2. Не-breaking изменения НЕ меняют версию, но фиксируются в changelog
3. Нумерация: v1, v2, v3 (без minor/patch)
4. Это гипотеза — если major-only не хватит при обкатке, добавим minor
```

### Breaking change (= новая версия)

```
Потребитель контракта СЛОМАЕТСЯ без изменений с его стороны:
  ✗ Удалено поле из выхода
  ✗ Переименовано поле
  ✗ Изменён тип поля (string → number)
  ✗ Добавлено обязательное поле на вход
  ✗ Изменён транспорт (HTTP → pipe)
  ✗ Изменён URL/endpoint
```

### Не breaking (= версия не меняется)

```
Потребитель контракта продолжит работать:
  ✓ Добавлено optional поле на выход
  ✓ Добавлено optional поле на вход (с дефолтом)
  ✓ Изменена реализация при том же контракте
  ✓ Изменено время отклика (если нет SLA)
```

### Формат при breaking change

```markdown
contract: [name] v[N]
breaking-change-from: v[N-1]
reason: [что изменилось]
changelog: [ссылка на запись]
```

### Changelog как система версионирования

Отдельная система не нужна. Changelog обязателен для каждого блока, RAG по changelog индексирует историю. Версия (v1, v2) — быстрый маркер для стыковки. Детали изменений — в changelog, доступны через RAG-запрос.

## 2.4 Каскадное обновление контрактов

При breaking change контракта — автоматическая генерация задач, ручное решение.

### Процесс

```
1. Breaking change в контракте блока A (v1 → v2)
       ↓
2. Оркестратор автоматически:
   - Находит все блоки, зависящие от A v1 (из BLOCKS.md)
   - Для каждого генерирует задачу на обновление
   - Оценивает сложность (переименование поля vs переделка логики)
   - Предлагает альтернативу: адаптер v1→v2 вместо обновления
       ↓
3. Человек решает:
   - Какие блоки обновлять сейчас, какие позже
   - Нужен ли адаптер (временный или постоянный)
   - Приоритет и порядок обновления
       ↓
4. Оркестратор генерирует промпты для агентов по решению человека
```

### Почему человек решает

```
Оркестратор видит:              Человек видит:
  - Граф зависимостей             - Бизнес-приоритеты
  - Технические последствия       - Цель проекта (GOAL.md)
  - Сложность изменений           - Сроки и ресурсы
  - Варианты решения              - Контекст пользователей

Оркестратор предлагает план → Человек выбирает → Оркестратор исполняет
```

## 2.5 Валидация стыков

Обязательный шаг оркестрации. Не факультативный.

### Что проверяется

```
Для каждой пары связанных блоков (A → B):

1. Формат данных
   - Выход A совместим со входом B?
   - Типы полей совпадают?
   - Обязательные поля выхода A покрывают обязательные поля входа B?

2. Транспорт
   - Выход A доступен по транспорту, который ожидает B?
   - HTTP → HTTP: ✓
   - stdout → HTTP: ✗ (нужен адаптер)

3. Версии
   - B работает с контрактом A v[N], а A уже v[N+1]?
   - Если да → каскадное обновление (см. 2.4)
```

### Когда проверяется

```
- При создании нового блока (до реализации)
- При breaking change контракта (до обновления зависимых)
- При сборке системы (перед деплоем)
- При диагностике проблем (что сломалось на стыке?)
```

## 2.6 Shared State (State Service)

Shared state — обычный блок с контрактом. Не особая сущность, не часть инфраструктуры.

### Почему обычный блок

```
На уровне управления:
  - Оркестратор знает, кто к state обращается
  - Оркестратор управляет доступом (кто читает, кто пишет)
  
На уровне реализации:
  - State Service = блок с контрактом
  - Реализация заменяема (SQLite → Postgres → бинарный движок)
  - Другие блоки обращаются к нему через стандартный стык
```

### Контракт State Service

```
Два режима в одном контракте (одна логическая ответственность):

1. Точный доступ (CRUD)
   POST   /data/{collection}         → создать запись
   GET    /data/{collection}/{id}    → получить по ID
   GET    /data/{collection}?filter= → фильтрация
   PUT    /data/{collection}/{id}    → обновить
   DELETE /data/{collection}/{id}    → удалить

2. Семантический поиск (векторный)
   POST   /search                    → поиск по смыслу
   POST   /index                     → индексация документа
```

### Реализация по умолчанию

```
Postgres + pgvector (через Supabase)

Путь:
  1. Self-hosted Supabase на финском сервере
     (Postgres + pgvector + REST API + дашборд, бесплатно)
  2. Фоллбэк: Supabase Cloud (от $25/мес)
  3. Суть: Supabase = Postgres + pgvector в обёртке с API

Почему Postgres + pgvector, а не "всё в вектор":
  - Векторная база не поддерживает точные запросы по ID
  - Нет транзакций (ACID)
  - Нет фильтрации по полям
  - Комбинированный подход: реляционный + векторный в одном блоке
```

### Масштабирование

```
Tier 2 (8-15 блоков):
  Один State Service блок — достаточно

Tier 3+ (40+ блоков):
  State Service разделяется на два блока:
    - DataStore (реляционный) — CRUD, транзакции
    - KnowledgeBase (векторный) — RAG, семантический поиск
  Контракты каждого уже содержатся в текущем едином контракте
  Разделение безболезненно
```

## 2.7 Граф зависимостей

Фиксируется в BLOCKS.md. Оркестратор читает его для валидации стыков и каскадных обновлений.

### Формат BLOCKS.md

```markdown
# BLOCKS — граф блоков проекта

## Блоки

| Блок | Контракт | Версия | Статус |
|------|----------|--------|--------|
| auth | contracts/auth.md | v1 | ✅ ready |
| billing | contracts/billing.md | v1 | 🔧 in progress |
| state-service | contracts/state.md | v1 | ✅ ready |

## Зависимости

auth → state-service v1
billing → state-service v1
billing → auth v1
notifications → auth v1
notifications → billing v1

## Слои

Слой 0 (инфраструктура): state-service
Слой 1 (ядро):            auth
Слой 2 (бизнес-логика):   billing, notifications
Слой 3 (интерфейс):       dashboard, api-gateway
```

### Слои

```
Блоки группируются в слои по уровню зависимости:

Слой 0: Инфраструктура — State Service, внешние API
         Ни от кого не зависит внутри проекта

Слой 1: Ядро — авторизация, базовые сервисы
         Зависит только от слоя 0

Слой 2: Бизнес-логика — основные функции
         Зависит от слоёв 0-1

Слой 3: Интерфейс — фронтенд, API-gateway
         Зависит от слоёв 0-2

Правило: блоки в одном слое НЕ зависят друг от друга.
Зависимости только сверху вниз (слой N → слой N-1).
Циклические зависимости запрещены.
```

## 2.8 GOAL для каждого блока

Каждый блок наследует свою часть цели проекта.

### Формат

```
В CONTRACT.md каждого блока — секция "Цель":

## Цель
Проектная цель (из GOAL.md): [общая цель]
Цель блока: [конкретно что этот блок вносит в достижение общей цели]
```

### Принцип

```
Цель блока:
  - Направляет реализацию (зачем это существует)
  - НЕ расширяет контракт (если цель не требует фичу — фичи нет)
  - Проверяется при ревью: блок выполняет свою цель?
  - Защищает от feature creep: если фича не ведёт к цели блока — она не нужна
```

## 2.9 Эвристики: когда дробить, когда объединять

### Дробить, если:

```
1. У блока ДВЕ причины для изменения
   (пример: блок и авторизует, и отправляет email → разделить)

2. Блок можно описать только через "и"
   ("делает авторизацию И отправку уведомлений" → два блока)

3. Часть блока нужна другим, а часть — нет
   (пример: парсинг данных нужен трём блокам, а визуализация — одному)

4. Внутри блока есть разные ритмы изменений
   (пример: бизнес-логика меняется часто, а работа с БД — редко)
```

### Объединять, если:

```
1. Два блока всегда меняются вместе
   (если изменение одного всегда требует изменения другого → один блок)

2. Блок не имеет смысла без другого
   (пример: "валидатор полей" без "формы" бессмысленен)

3. Разделение создаёт больше стыков, чем решает проблем
   (три блока по 10 строк с 6 стыками → один блок с двумя стыками)

4. Контракт между двумя блоками повторяет их внутреннюю структуру
   (если контракт стыка = пересказ реализации → это один блок)
```

### Красные флаги

```
⚠️ Блок зависит от 5+ других блоков
   → возможно, он делает слишком много

⚠️ Блок ни от кого не зависит и никто от него не зависит
   → зачем он существует?

⚠️ Описание блока требует слова "и" более одного раза
   → вероятно, нужно разделить

⚠️ Для изменения одной фичи нужно трогать 3+ блоков
   → неправильные границы, пересмотреть декомпозицию
```

## 2.10 Размер блока и capabilities исполнителя

Размер блока в контракте — не абсолютная величина. Он зависит от capabilities исполнителя.

```
Исполнитель                    Оптимальный размер блока
─────────────────────────────────────────────────────
DeepSeek / Cline (ранний)      Один модуль, 50–200 строк
Claude Code / Cursor            Целая страница/фича, 500–2000 строк
Человек + AI                    Любой (зависит от человека)
```

### Калибровка

Начни с мелких блоков (один компонент / один endpoint). Первые 5–10 контрактов — калибровочные: ты понимаешь, с чем справляется исполнитель, где он теряется, какой уровень детализации в контракте достаточен.

По мере калибровки — укрупняй. Если исполнитель стабильно справляется с компонентом за одну итерацию — дай ему страницу целиком.

### Процессный инвариант: контракт после фундамента

Операционный контракт пишется ПОСЛЕ того, как фундамент готов (API, адаптеры, миграции). Исполнитель получает готовый интерфейс с реальными данными, не обещание.

```
❌ Параллельно: бэкенд + контракт + фронт → конфликты, устаревшие спеки
✅ Последовательно: бэкенд → контракт → реализация → приёмка
```

→ Подробно: `patterns/contract-spec.md`, секция "Процессные инварианты"
