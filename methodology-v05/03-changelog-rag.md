## 3.1 Зачем два слоя документирования

Система порождает два типа артефактов: артефакты исполнения (код, бинарники, конфиги) и артефакты решений (changelog). Оба необходимы, но служат разным целям.

### Артефакт исполнения

```
Код, бинарник, конфигурация.
Отвечает на вопрос: ЧТО система делает прямо сейчас.
Содержит: текущее состояние реализации.
Не содержит: почему выбрано именно это решение,
             какие альтернативы рассматривались,
             что не работало раньше.
```

### Артефакт решений (changelog)

```
Структурированный лог.
Отвечает на вопрос: ПОЧЕМУ система такая, какая есть.
Содержит: решения, контекст, альтернативы, результаты.
Не содержит: текущее состояние кода.
```

### Почему код недостаточен

При традиционной разработке код — читаемый документ. Опытный разработчик восстанавливает намерение по реализации. В блочном кодинге это ломается по двум причинам:

**Бинарные блоки нечитаемы.** Когда содержимое блока — скомпилированный бинарник, код-ревью невозможно. Changelog становится единственным источником "почему так сделано". Без него бинарный блок — непрозрачная коробка без документации.

**AI не помнит между сессиями.** Каждая новая сессия AI-агента начинается с чистого листа. Без changelog агент не знает: какие решения приняты, что уже пробовали, что работает, а что нет. Changelog — мост между сессиями.

### Принцип из Манифеста

> Документирование — до, а не после. CHANGELOG пишется *до* изменения: что меняем, почему, какой ожидаемый результат.

Это не бюрократия. Это механизм, который:
- Фиксирует намерение до искажения результатом
- Создаёт контекст для любого исполнителя (человек или AI)
- Защищает от повторных ошибок (причина зафиксирована рядом с решением)

## 3.2 Формат changelog-записи

Changelog-запись создаётся в ДВА этапа: план (до работы) и факт (после).

### Этап 1: ПЛАН (до начала работы)

```markdown
## [block-name] | [YYYY-MM-DD] | [агент] | ПЛАН

### Цель блока
- [цель из CONTRACT.md — зачем этот блок существует]

### Что планируем сделать
- [конкретно что будет реализовано]

### Контракт
- Вход: contracts/[name].md v[N]
- Выход: [ожидаемое]

### Ожидаемые метрики
- [что будем проверять]
```

### Этап 2: ФАКТ (после завершения)

```markdown
## [block-name] | [YYYY-MM-DD] | [агент] | ФАКТ

### Сделано
- [конкретно что реализовано]

### Контракт
- Вход: соответствует contracts/[name].md v[N] ✅/❌
- Выход: [описание] ✅/❌

### Тесты (поведение)
- [вход X → выход Y]: ✅/❌
- [граничный случай]: ✅/❌
- [метрика скорости]: ✅/❌

### Отклонения от плана
- [если есть — что пошло не так и почему]
```

### Правила записи

```
1. ПЛАН пишется ДО начала работы — фиксирует намерение
2. ФАКТ пишется ПОСЛЕ завершения — фиксирует результат
3. Формат строго по шаблону — RAG индексирует по структуре
4. Тесты описывают ПОВЕДЕНИЕ (вход → выход), не реализацию
5. Отклонения от плана обязательны — это источник обучения
6. Если блок изменяется повторно — новая пара ПЛАН/ФАКТ
7. Одна запись = одна логическая единица работы (не день, не коммит)
```

### Гранулярность

Уровень **решений и поведения**, не редактирования текста.

```
✅ "Решили что shared state — обычный блок с контрактом, 
    не особая сущность. Причина: единообразие управления"
✅ "Блок auth: вход username+password → выход JWT token. 
    Тест: невалидный пароль → 401 за <50ms"
✅ "Отклонение: планировали HTTP, реализовали pipe — 
    HTTP добавлял 15ms латентности на внутреннем стыке"

❌ "Поправили абзац 3 в разделе 2"
❌ "Переименовали переменную"
❌ "Обновили зависимости"
```

## 3.3 Кто пишет changelog

Три автора, три уровня записей. Каждый пишет о своём уровне ответственности.

### Агент-исполнитель

```
Пишет: ПЛАН/ФАКТ по своему блоку
Когда: перед началом работы (ПЛАН), после завершения (ФАКТ)
Содержание:
  - Что собирается сделать / что сделал
  - Соответствие контракту
  - Результаты тестов (поведение)
  - Отклонения от плана
  - Обнаруженные проблемы

НЕ пишет:
  - Архитектурные решения
  - Бизнес-контекст
  - Решения по стыкам между блоками
```

### Оркестратор

```
Пишет: решения по стыкам, каскадные обновления, диагностику
Когда: при каждом решении (сразу, не в конце сессии)
Содержание:
  - Результат валидации стыков
  - Каскадные обновления контрактов
  - Адаптеры между блоками
  - Диагностика: какой блок сломался, почему
  - Статус блоков после сборки

НЕ пишет:
  - Детали реализации блоков
  - Бизнес-решения (это человек)
```

### Человек-архитектор

```
Пишет: бизнес-решения, ретроспективы, стратегические изменения
Когда: при принятии решения (фиксируется оркестратором по диктовке)
Содержание:
  - Почему строим именно это
  - Изменения в GOAL.md
  - Приоритеты и trade-off'ы
  - Ретроспективы (что работает, что нет)

НЕ пишет:
  - Технические детали реализации
```

### Правило фиксации

```
Решение принято → записать СРАЗУ.
Не "в конце сессии". Не "когда будет время". Не "запомню".
Что не записано — потеряно.
```

## 3.4 RAG-архитектура

### Проблема

Changelog растёт линейно с каждой сессией. Контекстное окно AI — фиксированное. При проекте длиннее 2-3 недель чтение полного changelog перестаёт помещаться в контекст или становится неэффективным (90% записей нерелевантны для текущей задачи).

### Решение

Вместо "загрузить весь changelog в контекст" — разбить знание на слои и подтягивать только релевантное через RAG (Retrieval-Augmented Generation).

```
БЕЗ RAG:
┌─────────────────────────────┐
│ AI Context Window            │
│ ┌─────────────────────────┐ │
│ │ Весь changelog (50k tok) │ │  ← Переполнение
│ │ Весь код проекта         │ │  ← Модель деградирует
│ │ Все контракты            │ │  
│ └─────────────────────────┘ │
└─────────────────────────────┘

С RAG:
┌─────────────────────────────┐
│ AI Context Window            │
│ ┌─────────────────────────┐ │
│ │ Текущая задача (500 tok) │ │
│ │ RAG: релевантные записи │ │  ← 2k токенов
│ │ RAG: verified facts      │ │  ← 500 токенов
│ │ Контракт блока (500 tok) │ │
│ └─────────────────────────┘ │  = ~4k вместо 50k+
└─────────────────────────────┘
         │
    ┌────▼────────────────────┐
    │   Слои знаний (RAG)     │
    │                         │
    │  📋 Changelog           │
    │  🧱 Code Blocks         │
    │  📐 Workflow            │
    │  📡 API Docs            │
    │  🐛 Known Issues        │
    │  ✅ Decisions           │
    └─────────────────────────┘
```

### Ключевая метрика

**<10k токенов контекста для любой задачи, независимо от размера проекта.**

Проект растёт → RAG-база растёт → контекст конкретной задачи НЕ растёт.

## 3.5 Слои знаний

Шесть слоёв, каждый с чётким назначением и триггером использования.

### Слой 1: Changelog (история изменений)

```
Источник: changelog-записи агентов, оркестратора, человека
Содержит: что менялось, почему, результаты, отклонения
Когда нужен:
  - Дебаг ("что менялось в этом блоке?")
  - Планирование ("какие проблемы были с этим модулем?")
  - Каскадное обновление ("кто зависит от этого контракта?")
Обновление: после каждой changelog-записи
```

### Слой 2: Code Blocks (блочная структура)

```
Источник: BLOCKS.md, контракты блоков
Содержит: что делает каждый блок, от чего зависит, статус
Когда нужен:
  - Агент начинает работу ("что делает мой блок, кто от него зависит?")
  - Оркестратор валидирует стыки
Обновление: при изменении структуры или контрактов
```

### Слой 3: Workflow (правила работы)

```
Источник: CLAUDE.md, project instructions, правила changelog
Содержит: как работать, формат записей, правила коммитов
Когда нужен: всегда — как базовый контекст для любого агента
Обновление: редко, при изменении процессов
```

### Слой 4: API Docs (документация зависимостей)

```
Источник: документация внешних API и сервисов
Содержит: endpoints, параметры, форматы, коды ошибок
Когда нужен: при работе с внешними зависимостями
Обновление: при изменении API
```

### Слой 5: Known Issues (известные проблемы)

```
Источник: секции "отклонения" и "проблемы" из changelog
Содержит: баги, workaround'ы, грабли на которые уже наступали
Когда нужен: при любом изменении — чтобы не повторить ошибку
Обновление: автоматически из changelog
```

### Слой 6: Decisions (архитектурные решения)

```
Источник: секции "решения" из changelog
Содержит: почему выбрали X а не Y, утверждённые подходы
Когда нужен: при архитектурных вопросах, при рефакторинге
Обновление: автоматически из changelog
```

### Связь между слоями

```
Слои 5 и 6 — производные от слоя 1 (Changelog).
RAG-индексатор парсит changelog-записи и автоматически
относит содержимое к нужному слою:

  ПЛАН/ФАКТ → Слой 1 (Changelog)
  Секция "Решения" → Слой 6 (Decisions)
  Секция "Отклонения" / "Проблемы" → Слой 5 (Known Issues)
  
Слои 2, 3, 4 индексируются из отдельных источников.
```


### На практике: Knowledge Stack

Шесть слоёв выше — теоретическая модель. На практике (MARS, 10K+ строк CHANGELOG, 33 контракта) кристаллизовалась конкретная реализация — паттерн Knowledge Stack:

**ADR как first-class citizens.** Слой 6 (Decisions) описан как "производный от CHANGELOG через RAG-парсинг". На практике это не работает при масштабе: решения тонут в техлоге. Решение — отдельная директория `decisions/NNN-*.md`, одно решение = один файл. Решения становятся самостоятельными документами, а не производными.

**Правило перенаправления.** Механизм, предотвращающий свалку "всё в CHANGELOG":

```
Пишу решение в CHANGELOG → СТОП → это ADR → выношу в decisions/
Пишу про клиента → СТОП → это дневник → выношу в PRODUCT_DIARY.md
Пишу про процесс → СТОП → обновить WORKFLOW
```

**PRODUCT_DIARY** — отдельный слой для бизнес/клиентского знания, не описанный в теоретической модели. Нужен когда есть клиенты и бизнес-контекст влияет на техническую работу.

**Статический контекст для агента (CLAUDE.md)** — не RAG-запрос, а курируемый файл, который агент читает в начале каждой сессии. Дополняет RAG: файл даёт базовый контекст (архитектура, verified facts, конвенции, запреты), RAG даёт детали по запросу.

→ Подробно: `patterns/knowledge-stack.md`, `patterns/agent-context.md`

## 3.6 Чанкинг и индексация

### Единица чанкинга

Чанк — одна логическая запись из changelog (пара ПЛАН/ФАКТ или отдельная сессия).

```
Почему не по абзацам: теряется контекст решения
Почему не по файлам: файл может содержать 100 записей
Почему по записям: одна запись = одно решение = минимальная 
                   единица знания со смыслом
```

### Метаданные чанка

```json
{
  "block":       "auth",              // имя блока (или "project" для общих)
  "date":        "2026-02-18",       // дата записи
  "author":      "agent-coder",      // кто написал
  "type":        "fact",             // plan | fact | decision | issue
  "layer":       "changelog",        // к какому слою относится
  "source_type": "changelog",        // категория источника (см. 3.7)
  "contract":    "auth.md v2",       // к какому контракту (если есть)
  "tags":        ["breaking-change", "http", "jwt"]  // ключевые слова
}
```

Поле `source_type` — категория документа-источника. Используется для маршрутизации запросов (см. 3.7 "Document-aware routing"). Четыре значения:

```
"methodology"      — формальные определения, принципы, структура
                     (файлы methodology-v05/*, MANIFESTO, GOAL)
"decisions"        — принятые решения, их контекст и альтернативы
                     (секции Decisions/Решения из CHANGELOG)
"implementation"   — что конкретно сделано, результаты, факты
                     (секции ФАКТ/Сделано/Verified из CHANGELOG)
"issues"           — проблемы, отклонения, workaround'ы
                     (секции Known Issues/Отклонения из CHANGELOG)
```

"verified"         — факты, проверенные вручную, которые НЕЛЬЗЯ менять
                     (секции Verified из CHANGELOG, CLAUDE.md)
                     → patterns/verified-facts.md

### Теги

Автоматические (извлекаются из записи): имена блоков, версии контрактов, типы транспорта.

Ручные (необязательные): для неочевидных связей, которые автоматический парсер не найдёт.

### Формат запроса

```
Кто запрашивает → какие слои → какой контекст

Агент (перед работой):
  "changelog + known-issues для блока auth"
  → Последние записи по блоку + известные проблемы

Оркестратор (при диагностике):
  "все изменения за сегодня + decisions по shared state"
  → Хронология дня + архитектурные решения по теме

Человек (статус):
  "что реально работает прямо сейчас"
  → Последние ФАКТ-записи со статусом ✅
```

## 3.7 Интеграция в рабочий процесс

### Document-aware routing (маршрутизация запросов)

Не все запросы к RAG одинаковы. Запрос "как определить границу блока" и запрос "почему выбрали Go для ботов" — принципиально разные по типу знания: первый ищет **определение** (methodology), второй — **решение** (decisions). Если искать оба в общем пуле без фильтрации, результаты зашумлены нерелевантными чанками.

**Принцип: перед RAG-запросом определи категорию знаний.**

```
Шаг 1: Классифицируй запрос → source_type
Шаг 2: Ограничь поиск нужной категорией
Шаг 3: Ищи по score внутри категории
```

Четыре категории (совпадают с `source_type` в метаданных чанка, см. 3.6):

```
methodology    — "что такое X", "как определяется Y", "какой формат Z"
                 → искать в methodology-v05/*, MANIFESTO, GOAL
                 
decisions      — "почему выбрали X", "какие альтернативы были"
                 → искать в секциях Decisions/Решения CHANGELOG
                 
implementation — "что реально сделано", "какой результат", "статус"
                 → искать в секциях ФАКТ/Сделано/Verified CHANGELOG
                 
issues         — "какие проблемы были", "что ломалось", "workaround"
                 → искать в секциях Known Issues/Отклонения CHANGELOG
```

### Правило для агента (промптовый уровень)

```
Перед RAG-запросом задай себе вопрос:
  "Я ищу определение, решение, факт или проблему?"

  Определение → source_type: methodology
  Решение     → source_type: decisions
  Факт        → source_type: implementation
  Проблему    → source_type: issues
  Не уверен   → без фильтра (общий поиск)
```

Это не требует изменений в коде RAG — достаточно добавить `source_type` в метаданные при индексации и передавать как фильтр при запросе. Для Tier 1 правило работает на уровне промпта без фильтрации (агент сам выбирает где искать). Для Tier 2+ — metadata-фильтр в RAG API.

### Примеры маршрутизации

```
Запрос: "как определить границу блока"
  → Категория: methodology
  → Поиск в: methodology-v05/02-decomposition.md
  → НЕ искать в: CHANGELOG (там решения по конкретным блокам, не определения)

Запрос: "почему выбрали Go для ботов"
  → Категория: decisions
  → Поиск в: CHANGELOG секции Decisions
  → НЕ искать в: methodology-v05/ (там нет конкретных технологических решений)

Запрос: "работает ли бот Павла"
  → Категория: implementation
  → Поиск в: CHANGELOG секции ФАКТ/Verified
  → НЕ искать в: methodology-v05/ (там нет статуса конкретных проектов)

Запрос: "какие проблемы были с RAG-индексацией"
  → Категория: issues
  → Поиск в: CHANGELOG секции Known Issues
  → НЕ искать в: methodology-v05/ (там описание как должно быть, не что ломалось)
```

### Связь с архитектурой MSA

Паттерн document-aware routing вдохновлён Memory Sparse Attention (EverMind, 2026): вместо полного перебора всех чанков — сначала маршрутизация к нужному подмножеству документов, затем поиск внутри него. MSA делает это на уровне нейронной архитектуры (routing keys + Top-k selection), мы — на уровне метаданных и промптовых правил. Принцип один: иерархический отбор эффективнее плоского перебора.

### Агент перед работой — автоматический RAG-запрос

Перед началом работы агент получает (или запрашивает) контекст из RAG:

```
1. Контракт своего блока (из файла, не из RAG)
2. RAG → changelog по своему блоку (последние 3-5 записей)
       [source_type: implementation]
3. RAG → known issues по своему блоку и зависимостям
       [source_type: issues]
4. RAG → verified facts, которые НЕЛЬЗЯ менять
       [source_type: implementation, фильтр: verified]
```

Это заменяет чтение всего проекта. Агент получает ~2-4k токенов контекста вместо 50k+.

### Оркестратор — поиск по решениям

```
Сценарий: нужно понять почему выбран HTTP а не pipe для блока X

Без RAG: перечитать весь changelog глазами (400+ строк)
С RAG:   запрос "транспорт блок X http pipe" 
         [source_type: decisions]
         → релевантная запись за 2 секунды
```

### Человек — "что реально работает"

```
Сценарий: Гриша вернулся после выходных, нужно понять статус

Без RAG: прочитать changelog за 2 дня + спросить Claude
С RAG:   запрос "статус блоков за последние 3 дня" 
         [source_type: implementation]
         → сводка
```

### Шаблон промпта для агента с RAG

> **На практике** этот шаблон реализуется через статический файл CLAUDE.md
> (паттерн agent-context), который агент читает в начале каждой сессии.
> RAG-запросы дополняют статический контекст деталями по запросу.
> → `patterns/agent-context.md`


```markdown
# Контекст задачи

## Контракт блока
[содержимое CONTRACT.md]

## Последние изменения (из RAG)
[запрос: changelog по блоку, source_type: implementation, 3-5 записей]

## Известные проблемы (из RAG)
[запрос: known issues по блоку, source_type: issues]

## Что нельзя менять (из RAG)
[запрос: verified facts, source_type: implementation, фильтр: verified]

## Задача
[конкретная задача от оркестратора]
```

## 3.8 Масштабирование по тирам

### Tier 1 (1-5 блоков, лендинг)

```
Changelog: один файл CHANGELOG.md
RAG: не нужен — файл помещается в контекст (~100-200 строк)
Поиск: grep / Ctrl+F достаточно
Document-aware routing: на уровне промпта (агент сам решает где искать)
```

### Tier 2 (8-15 блоков, типичный проект)

```
Changelog: один файл CHANGELOG.md
RAG: обязателен — файл перестаёт помещаться в контекст (~500-2000 строк)
Поиск: RAG с индексацией по слоям
Индекс: единый, на одном сервере
Document-aware routing: metadata-фильтр по source_type в RAG API
```

### Tier 3 (40+ блоков, enterprise)

```
Changelog: распределённый — каждый блок имеет свой changelog
           + общий проектный changelog (решения уровня системы)
RAG: обязателен, с агрегацией из нескольких источников
Поиск: RAG по всем changelog'ам + фильтрация по блоку/дате/автору
Индекс: может потребоваться распределённый (по командам / подсистемам)
Document-aware routing: обязателен — без него шум в результатах растёт квадратично
```

### Переход между тирами

```
Tier 1 → Tier 2:
  Триггер: changelog перестаёт помещаться в контекст AI (~500 строк)
  Действие: развернуть RAG-индекс, начать индексировать changelog
  
Tier 2 → Tier 3:
  Триггер: один changelog содержит записи 40+ блоков, поиск медленный
  Действие: разделить на per-block changelog'и + общий
  Триггер 2: команда больше 3 человек — нужна агрегация записей
```

## 3.9 Инфраструктура

### Технический стек

```
Индексация:   парсинг markdown → чанки → эмбеддинги → векторная БД
Поиск:        семантический (по смыслу) + фильтрация (по метаданным)
MCP tools:    search_changelog, search_known_issues, search_decisions
Обновление:   при записи в changelog (триггер) или по расписанию
```

### Принцип заменяемости

Конкретные инструменты (модель эмбеддингов, векторная БД, способ индексации) — заменяемы. Методология определяет:

```
ЧТО:   6 слоёв знаний, формат чанка, метаданные, интерфейс запроса
НЕ ЧТО: конкретная БД, конкретная модель, конкретный хостинг

Сегодня: скрипт парсинга + простой векторный поиск на сервере
Завтра:  managed RAG-сервис или встроенный в IDE
Контракт не меняется: запрос по ключевым словам → релевантные записи
```

### Триггеры обновления индекса

```
1. Post-commit hook: новая запись в changelog → переиндексация
2. По запросу: оркестратор запрашивает "обнови индекс"
3. По расписанию: ежедневно (фоллбэк, если хуки не сработали)
```

### Минимальный MVP

```
Для проекта на Tier 2:
1. Парсер: markdown → чанки (по секциям ## и ###)
2. Индекс: эмбеддинги через модель + векторная БД
3. MCP tool: search_changelog(query) → top-5 релевантных записей
4. Тест: "найди решение про shared state" → запись от 18.02 ✅

Этого достаточно для начала. Остальные слои добавляются итерационно.
```

## 3.10 Антипаттерны

### ❌ "Запишу потом"

```
Проблема: решение принято в обсуждении, запись откладывается
Результат: через 2 часа контекст решения забыт, 
           запись неполная или отсутствует
Правило: записать СРАЗУ, даже черновик лучше пустоты
```

### ❌ "Это очевидно, не буду записывать"

```
Проблема: решение кажется очевидным сейчас
Результат: через 2 недели (или для нового агента) — непонятно 
           почему выбран этот подход
Правило: записывать ВСЁ, включая "очевидное". 
         Особенно — альтернативы, которые отвергли (и почему)
```

### ❌ "Changelog = git log"

```
Проблема: подменяют changelog списком коммитов
Результат: "fix: typo", "update deps", "refactor" — 
           нулевая ценность для понимания решений
Правило: changelog фиксирует РЕШЕНИЯ и ПОВЕДЕНИЕ, 
         git log фиксирует ИЗМЕНЕНИЯ КОДА. Это разные вещи.
```

### ❌ "Один changelog на всё"

```
Проблема: в одном файле — архитектурные решения, 
          баг-фиксы, бизнес-логика, инфраструктура
Результат: при поиске 90% записей нерелевантны
Правило: для Tier 1-2 — один файл, но с тегами для RAG.
         Для Tier 3 — разделять по блокам
```

### ❌ "RAG решит все проблемы"

```
Проблема: надежда что RAG компенсирует плохой changelog
Результат: garbage in → garbage out. Плохие записи → плохой поиск
Правило: RAG усиливает хороший changelog, 
         не спасает плохой. Качество записей — первичнее RAG
```

### ❌ "Всё в одну кучу без маршрутизации"

```
Проблема: все чанки (определения, решения, факты, проблемы) 
          лежат в одной коллекции без source_type, 
          поиск идёт по всему пулу
Результат: запрос "как определить границу блока" возвращает 
           записи из CHANGELOG о конкретных блоках вместо 
           формального определения из methodology-v05/
Правило: добавить source_type в метаданные при индексации. 
         Перед запросом определить категорию знаний (methodology / 
         decisions / implementation / issues). Фильтровать ДО 
         семантического поиска, не после.
         См. document-aware routing в секции 3.7.
```
