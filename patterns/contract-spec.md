# Паттерн: Contract Spec — операционный контракт для AI-оператора

## Проблема

Методология определяет контракт как интерфейс между блоками: вход, выход, транспорт, ограничения. Этого достаточно для оркестрации (проверка стыков, версионирование, каскадные обновления), но недостаточно для AI-оператора, который должен реализовать блок.

AI-оператор (Claude Code, Cline, или человек-разработчик с AI) нуждается в другом уровне детализации:
- Что уже готово? (API, адаптеры, миграции — с реальными данными, не абстракциями)
- Что конкретно сделать? (пошагово по компонентам, с примерами кода)
- Что НЕ трогать? (файлы, паттерны, области другого оператора)
- Как проверить? (конкретные сценарии с конкретными данными)

Без этого оператор интерпретирует абстрактное описание по-своему: лезет в чужие файлы, ломает работающий код, делает "как понял" вместо "как задумано".

## Когда применять

- AI-оператор реализует блок (Claude Code, Cline, или человек + AI)
- Несколько операторов работают параллельно (нужны границы "не трогать")
- Блок имеет фронтенд (UI-спецификации требуют конкретики)
- Бэкенд/API уже готов и оператор должен работать с реальным интерфейсом

Не нужен для:
- Одиночной работы архитектора (достаточно контракта из блока 02)
- Инфраструктурных задач без оператора
- Прототипирования (слишком рано для формализации)

## Решение

Операционный контракт — это расширение архитектурного контракта конкретикой для исполнителя. Не заменяет контракт из блока 02 методологии, а дополняет его.

### Структура: шапка + шесть секций

#### Шапка (обязательно)

```markdown
# КОНТРАКТ: [название]

**Блок:** идентификатор
**Слой:** Backend / Frontend / Infra / Data
**Файлы:** какие создать, какие изменить
**Зависимости:** что должно быть готово до начала
**Дедлайн:** дата (если есть)
```

Шапка — для быстрого скана. Оператор за 10 секунд понимает: что за блок, какой слой, какие файлы, что должно быть готово.

#### §1 Что уже готово (обязательно)

Описание фундамента, на котором оператор строит. Конкретные API endpoints с реальными JSON-ответами, готовые адаптеры, миграции, конфиг.

Принцип: **оператор не гадает что приходит — он видит конкретный JSON.** Не "API возвращает объект робота", а полный JSON с реальными данными из продакшена или тестовой среды.

```markdown
## 1. Что уже готово

API GET `/api/v2/devices` возвращает:

\```json
{
  "id": "uuid-пример",
  "vendor_id": "8110M5322060001",
  "status": "working",
  "battery": 84
}
\```

Через adapter (api.js) в компонент приходит как:
\```js
robot.status       // string
robot.battery      // number
\```
```

#### §2 Коды и таблицы маппинга (если есть)

Таблицы соответствий: код → отображение, статус → стиль, условие → поведение UI. Чем точнее таблица, тем меньше интерпретаций.

```markdown
## 2. Маппинг

| Код | severity | Отображение | Стиль |
|-----|----------|-------------|-------|
| LIVE | none | Статус как есть | Обычная |
| STALE | info | + «Данные от ЧЧ:ММ» | Обычная |
| OFFLINE | warning | «Не в сети» | Приглушённая |
```

Не обязательна для каждого контракта — только когда есть маппинг состояний, кодов, или таблицы решений.

#### §3 Что нужно сделать (обязательно)

Пошагово, по компонентам/модулям. Каждый пункт с примером кода или псевдокода. Не абстракции ("сделай красиво"), а конкретные правила.

Для фронтенда:
- §3.1 Новый компонент — props, логика рендера, примеры JSX
- §3.2 Изменения в существующем — что менять, как
- §3.N Сортировка, фильтры, стили — конкретные правила с кодом

Для бэкенда:
- §3.1 Новый endpoint — сигнатура, логика, примеры запроса/ответа
- §3.2 Изменения в модели — какие поля добавить, как мигрировать
- §3.N Интеграция — с какими сервисами, через какой протокол

Принцип: **если можно показать код или псевдокод — покажи.** "opacity-60 если dim_card=true" лучше чем "приглушить карточку".

#### §4 Что НЕ делать (обязательно)

Явный список запретов. Это **критически важная секция** — она предотвращает конфликты между операторами и защищает работающий код.

```markdown
## 4. Чего НЕ делать

- **НЕ трогать** api.js — адаптер уже обновлён
- **НЕ трогать** FreshnessBadge.jsx — остаётся для обратной совместимости
- **НЕ вычислять** ситуацию на фронте — всё приходит от API
- **НЕ добавлять** новые API-запросы — всё уже в GET /api/v2/devices
- **НЕ менять** стили других страниц
```

Три категории запретов:
1. **Файлы** — конкретные файлы/директории, которые не трогать
2. **Паттерны** — подходы, которые не применять ("не вычислять на фронте")
3. **Области** — части системы, принадлежащие другому оператору или другой волне

Если §4 пустой — контракт неполный. У любого блока есть границы.

#### §5 Тест приёмки (обязательно)

Конкретные сценарии с конкретными данными. Не "проверь что работает", а пошаговый сценарий с именами сущностей из реальной/тестовой среды.

```markdown
## 5. Тест приёмки

1. Открой страницу Роботы
2. Найди CC1 402 (811115320060402) — DATA_DEGRADED
3. Проверь: карточка приглушена + жёлтая полоса слева
4. Hover на бейдж → tooltip "Подключите робот к сети"
5. Найди FlashBot2025 — LONG_OFFLINE
6. Проверь: сильно приглушена, «Нет данных с 24.04»
7. Сортировка: LIVE-роботы сверху, NEVER_SEEN внизу
```

Принцип: **тест приёмки = чеклист для ревью.** Оркестратор (или архитектор) идёт по пунктам, каждый — pass/fail. Если все pass — блок принят.

#### §6 Файлы для справки (опционально)

Ссылки на спеки, документацию, модели данных, код evaluator — всё, что оператору может понадобиться для понимания контекста.

```markdown
## 6. Справка

- `docs/SITUATION_MODEL.md` — спецификация ситуаций
- `api/v2/evaluator.js` — код evaluator (для понимания логики)
- `docs/decisions/002-situation-model.md` — ADR: почему именно такая модель
```

### Процессные инварианты

**1. Контракт пишется ПОСЛЕ фундамента**

Сначала API/адаптер/миграция → потом контракт. Оператор получает готовый интерфейс с реальными данными, не обещание. Это убивает целый класс ошибок: "API изменился после написания контракта → контракт устарел → оператор делает по устаревшей спеке".

```
❌ Параллельно: бэкенд + контракт + фронт → конфликты, устаревшие спеки
✅ Последовательно: бэкенд → контракт → фронт → приёмка
```

**2. Размер блока зависит от capabilities исполнителя**

Не все AI-агенты одинаково способны. Размер блока в контракте адаптируется:

| Исполнитель | Размер блока | Пример |
|-------------|-------------|--------|
| DeepSeek / Cline (ранний) | Один модуль, 50–200 строк | "Создай компонент SituationBadge" |
| Claude Code / Cursor | Целая страница/фича, 500–2000 строк | "Переделай страницу Роботы под ситуационную модель" |
| Человек + AI | Любой | Зависит от человека |

Эволюция MARS: wave 1–6 — мелкие блоки (один компонент), wave 7–10 — крупные (целая страница). По мере роста доверия к исполнителю и зрелости инструмента блоки укрупняются.

Принцип: **начни с мелких блоков, укрупняй по мере калибровки.** Первые 5–10 контрактов — маленькие (один компонент/endpoint). Когда калибровка пройдена — один контракт на фичу.

**3. Граница ответственности — явная**

Контракт чётко разделяет: что сделано (архитектор/оркестратор) vs что делать (оператор). Пограничные файлы (api.js, shared utils) — явно указать владельца в §4.

```
❌ "api.js" (неясно — чей?)
✅ "api.js — НЕ ТРОГАТЬ, уже обновлён оркестратором"
```

### Связь с архитектурным контрактом (блок 02)

Два уровня контрактов дополняют друг друга:

| | Архитектурный (блок 02) | Операционный (этот паттерн) |
|---|---|---|
| **Для кого** | Оркестратор | AI-оператор |
| **Описывает** | Интерфейс блока | Спецификацию работы |
| **Содержит** | Вход/выход/транспорт/ограничения | §1–§6: что готово, что делать, что не трогать |
| **Когда** | При проектировании | При реализации |
| **Версионируется** | v1, v2 (breaking changes) | Версия контракта = версия работы |

Операционный контракт ссылается на архитектурный ("контракт блока auth v2"), но добавляет конкретику для исполнителя.

## Антипаттерны

### ❌ "Сделай красиво"
Субъективная инструкция → 100 интерпретаций. Вместо этого: конкретные CSS-классы, opacity-значения, цвета, отступы.

### ❌ §1 без реальных данных
"API возвращает объект робота" → оператор фантазирует о формате. Вместо этого: полный JSON из реальной/тестовой среды.

### ❌ Отсутствие §4
Оператор полезет в файлы другого оператора → конфликт. §4 — обязательная секция, не опциональная.

### ❌ Абстрактный тест приёмки
"Проверь что работает" ничего не проверяет. Вместо этого: конкретная сущность, конкретное ожидание, конкретный шаг.

### ❌ Контракт до бэкенда
API меняется → контракт устарел → оператор делает по мёртвой спеке → конфликт при приёмке. Контракт — ПОСЛЕ готового API.

### ❌ Одинаковый размер блоков для всех исполнителей
DeepSeek теряется в контракте на 2000 строк. Claude Code скучает на контракте в 50 строк. Калибруй размер под capabilities.

## Происхождение

Паттерн выведен из MARS (апрель 2026): 33 контракта, 10 волн, эволюция от мелких блоков (DeepSeek/Cline) к крупным (Claude Code). Формализован в CONTRACT_GUIDE.md. Эталонный контракт: wave-10/33-robots-page-situation.md.

Ключевой инсайт: контракт блока для оркестрации (блок 02) и контракт для исполнителя — разные документы с разными аудиториями. Первый отвечает "что этот блок делает в системе", второй — "как этот блок реализовать".
