# Паттерн: Verified Facts — защита проверенных фактов

## Проблема

AI-агент не различает "работает случайно" и "проверено вручную и гарантированно работает". Без явного маркера агент может "улучшить" или "отрефакторить" компонент, который прошёл ручную верификацию — и сломать его.

Проблема обостряется при:
- Нескольких операторах (один проверил, другой сломал)
- Длинных проектах (проверки из прошлого месяца забыты)
- Внешних интеграциях (HMAC-подписи, OAuth-токены, API-конвенции вендоров — выверены методом проб и ошибок, выглядят "неправильно" для агента)

Цена ошибки высокая: восстановление verified-факта требует повторной ручной проверки, часто с участием внешних систем.

## Когда применять

- Проект имеет внешние интеграции (API вендоров, платёжки, auth-провайдеры)
- AI-оператор работает с кодом, который уже проверен в продакшене
- Есть компоненты, рефакторинг которых дороже чем оставить как есть
- В команде больше одного участника (включая AI-операторов)

## Решение

Три механизма защиты, каждый на своём уровне.

### 1. Секция Verified в CHANGELOG

Каждая запись CHANGELOG может содержать секцию `Verified` — список фактов, проверенных вручную в этой сессии.

```markdown
### Verified (НЕ ЛОМАТЬ)
- **PUDU HMAC:** path WITH query params включая `?` — проверено на 3 связках ключей
- **React.lazy():** все страницы через Suspense — одна битая не роняет всё
- **MT серия:** water_stage=0 это НОРМА для сухой уборки, не баг
- **InteractiveMap:** area/edge/forbidden/virtual_wall для 6 из 8 PUDU роботов
```

Формат каждого факта: **что** + **как проверено** (или почему это важно). Голый факт без контекста ("HMAC работает") бесполезен — агент не поймёт, что именно нельзя трогать.

Verified-записи в CHANGELOG — append-only. Не удалять, не редактировать. Факт может быть superseded новым verified-фактом, но не стёрт.

### 2. Секция Verified в CLAUDE.md

Самые критичные verified-факты дублируются в `CLAUDE.md` (паттерн [agent-context.md](agent-context.md), секция 3). Это гарантирует, что агент увидит их в начале каждой сессии без RAG-запроса.

```markdown
## Verified факты (НЕ ЛОМАТЬ)

- **PUDU HMAC:** path WITH query params. Менять = ломать auth.
- **api.js адаптер:** владелец — Гриша. НЕ менять без согласования.
```

Критерий попадания в CLAUDE.md: факт, нарушение которого ломает систему или интеграцию. Не все verified-факты — только критичные.

### 3. RAG-индексация с типом `verified`

Verified-записи индексируются в RAG с `source_type: verified` (или как часть `implementation` с тегом). Это позволяет оркестратору перед генерацией контракта запросить:

```
"verified факты для модуля api/connectors/pudu.js"
```

и вставить найденные в §4 контракта ("Что НЕ делать").

### Жизненный цикл verified-факта

```
1. Ручная проверка → запись в CHANGELOG (Verified)
     ↓
2. Оценка критичности:
   - Критичный (ломает систему) → добавить в CLAUDE.md
   - Важный (нарушение = баг) → оставить в CHANGELOG, индексировать в RAG
   - Информационный (полезно знать) → только CHANGELOG
     ↓
3. При генерации контракта:
   - Оркестратор проверяет RAG: есть ли verified-факты для затрагиваемых файлов
   - Найденные факты → §4 контракта ("НЕ трогать")
     ↓
4. При изменении verified-компонента:
   - Новая проверка → новый verified-факт (supersedes старый)
   - Старый факт НЕ удаляется из CHANGELOG
```

### Категории verified-фактов

| Категория | Пример | Риск нарушения |
|-----------|--------|---------------|
| Интеграции | HMAC-подпись: path WITH query params | Auth ломается |
| Данные | MT серия: water_stage=0 = норма | Ложные алерты |
| UI | React.lazy + Suspense = изоляция ошибок | Одна страница роняет всё |
| Владение | api.js — владелец Гриша | Конфликт при мерже |
| Бизнес-логика | FlashBot = delivery, вода не применима | Неправильные метрики |

### Автоматическая защита (опционально)

Git pre-commit hook, который предупреждает при изменении файлов с verified-фактами:

```bash
# Псевдокод
changed_files = git diff --cached --name-only
verified_files = grep -r "НЕ ЛОМАТЬ" CLAUDE.md | extract_paths

for file in changed_files:
  if file in verified_files:
    warn "⚠️ Изменён файл с verified-фактом: $file"
    warn "Проверь CLAUDE.md секцию Verified"
```

Не блокирует — предупреждает. Иногда изменение verified-файла необходимо, но оно должно быть осознанным.

## Связь с методологией

Паттерн развивает два блока методологии v0.5:

**Блок 03 (Changelog + RAG):**

- Принцип "verified-факт имеет приоритет над любым автоматическим решением" — новый, в блоке 03 отсутствует.

**Блок 04 (Валидация):**
- Трёхуровневая валидация (sanity, контрактная, интеграционная) проверяет корректность НОВОЙ работы. Verified facts защищают СТАРУЮ проверенную работу от разрушения новой. Это комплементарные механизмы.

**Связь с другими паттернами:**

## Антипаттерны

### ❌ "Всё verified"
Если каждая строка — verified, ничего не verified. Маркер теряет смысл. Verified — только то, что проверено ВРУЧНУЮ и нарушение чего имеет конкретные последствия.

### ❌ "Verified без контекста"
"HMAC работает" — бесполезно. "HMAC: path WITH query params включая `?` — проверено на 3 связках ключей" — полезно. Агент должен понимать ЧТО ИМЕННО нельзя менять.

### ❌ "Verified-факты только в CHANGELOG"
CHANGELOG растёт, факты тонут. Критичные факты — в CLAUDE.md (агент видит каждую сессию). Остальные — в RAG (находятся по запросу). Только CHANGELOG — недостаточно.

### ❌ "Удаление старых verified"
Verified в CHANGELOG — append-only. Факт может стать неактуальным (superseded), но не удалённым. История верификаций — ценный контекст для понимания эволюции системы.

### ❌ "Агент сам решает что verified"
Verified — результат ручной проверки человеком. Агент может предложить пометить что-то как verified, но решение принимает оркестратор/архитектор. Автоматическая маркировка обесценивает статус.

## Происхождение

Паттерн выведен из MARS (февраль–апрель 2026). Ключевые эпизоды:
- PUDU HMAC: агент трижды "фиксил" подпись, каждый раз ломая auth. После добавления в Verified + CLAUDE.md — ноль инцидентов.
- MT серия water_stage=0: агент генерировал алерты на "нулевую воду" для роботов сухой уборки. Verified-факт "water_stage=0 = норма для MT" устранил ложные срабатывания.
- api.js адаптер: два оператора одновременно правили файл → merge conflict → баги. Verified-владение ("владелец — Гриша, НЕ менять") устранило конфликт.
