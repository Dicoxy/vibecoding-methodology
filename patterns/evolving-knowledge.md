## Что это (короткая формулировка)

Паттерн **"Эволюционирующее знание"** — структура для хранения знания, которое **продукт накапливает о своём домене** в процессе работы. Не статичная база данных, не append-only лог. **Живая структура, в которой текущее состояние понимания обновляется**, а история сохраняется через git.

Главный признак: brain хранит **"где мы сейчас"**, а не **"что мы изменили"**. Это отличает его от CHANGELOG.

## Когда применять

✅ **Применять**, если у проекта есть:

- **Домен** — сущности реального мира, с которыми продукт работает (клиенты, продукты, заявки, документы).
- **Пользователи**, которые работают с продуктом долго (не разовые посетители).
- **Накапливаемое знание** — продукт со временем "узнаёт" больше про свой домен (новые сущности, новые связи, новые правила).

❌ **Не применять**, если:

- Продукт — одноразовый скрипт или утилита.
- Нет домена в нужном смысле (например, чистая инфраструктура без бизнес-сущностей).
- У продукта нет постоянных пользователей.

## Базовая структура

```
brain/
├── BRAIN_CONFIG.md         ← правила игры конкретного проекта
├── _index.md               ← навигация (curator поддерживает)
├── _curator_log.md         ← журнал переосмыслений
│
├── entities/               ← живые сущности с ТЕКУЩИМ состоянием
│   └── ...                 (структура зависит от домена проекта)
│
├── concepts/               ← понятия, специфичные для проекта
│   └── ...
│
├── decisions/              ← хронология значимых решений
│   └── YYYY-MM-DD-*.md
│
└── ideas/                  ← открытые циклы (по желанию)
    └── _index.md
```

**Формат хранения** — markdown с wikilinks `[[name]]`. Совместимо с Obsidian.app, но Obsidian не обязателен.

**Поверх markdown** — Qdrant как кэш для быстрого поиска **только если объём данных оправдывает**. Markdown остаётся source of truth.

## Главное правило: brain не конкурирует с CHANGELOG

| | CHANGELOG | Brain |
|---|---|---|
| **Природа** | Append-only | Мутабельный |
| **Что хранит** | "Что мы изменили" | "Где мы сейчас" |
| **Аудитория** | Разработчики, для аудита истории | Пользователи продукта + AI-агенты, для текущей работы |
| **Срок жизни записи** | Бесконечный | Пока актуально |
| **Обновление** | Только дозапись | Перезапись с сохранением в git |

В одном проекте оба существуют. Они **дополняют**, не **заменяют**.

## Curator — обязательная роль

Brain без curator-а **деградирует за месяц**. Появляются дубли, противоречия, orphan-сущности, устаревшие записи, на которые никто не ссылается.

**На старте** curator — это архитектор + Claude в конце каждой сессии вручную. Достаточно для начала, не требует инфраструктуры.

**В перспективе** curator — отдельный AI-агент, потенциально многопайплайновый и с дообученной LLM. Это направление развития, не сегодняшняя задача. См. [`brain/ideas/2026-05-03-curator-as-multipipeline-agent.md`](../../projects/vibecoding/brain/ideas/2026-05-03-curator-as-multipipeline-agent.md).

## Как внедрять в существующий проект

Через **протокол** — отдельный документ, не часть этого паттерна:

→ [`protocols/embed-existing-project.md`](../protocols/embed-existing-project.md)

Не пытаться внедрить brain "с разбегу". Протокол — четыре этапа с обязательной паузой. Это не бюрократия, это страховка от перепрыгивания.

## Особый случай: продукт-фабрика с двойным brain

У некоторых продуктов brain существует на **двух уровнях одновременно**:

- **Уровень А** — продукт как мой проект-в-разработке (свой brain про развитие).
- **Уровень B** — каждый клиентский инстанс продукта (свой brain про домен клиента).

Это **отдельная конфигурация паттерна**, не частный случай. У каждого уровня — свой curator, свой ритм обновления, своя аудитория.

Пример — КИБА. См. [`brain/ideas/2026-05-03-kiba-double-brain.md`](../../projects/vibecoding/brain/ideas/2026-05-03-kiba-double-brain.md).

Этот случай не покрывается базовой структурой выше — будет отдельной развёрткой паттерна, когда дойдём до создания brain КИБА.

## Связь с другими паттернами

**[Knowledge Stack](knowledge-stack.md)** — смежный, не дублирует.
- Knowledge Stack: знание о **процессе разработки** (CHANGELOG, ADR, CLAUDE.md). Это для людей, строящих продукт.
- Evolving Knowledge: знание о **поведении продукта** (brain, который продукт использует в рантайме). Это для пользователей и AI-агентов *в* продукте.
- В одном проекте применяются **оба**.

**[Verified Facts](verified-facts.md)** — частный случай.
- Verified-факты домена (не кода) живут в `entities/` brain-а проекта, как часть текущего состояния понимания.

**[Agent Context](agent-context.md)** — смежный.
- Agent Context (CLAUDE.md) — для агента, который **строит код**.
- Brain — для агента, который **обслуживает рантайм продукта**.

**[Contract Spec](contract-spec.md)** — независим.
- Контракты определяют, как блоки общаются между собой. Brain — это контент, на котором блоки работают. Разные оси.

## Что НЕ покрыто этой заглушкой (полный текст напишется после обкатки)

- Конкретный шаблон `BRAIN_CONFIG.md` для разных типов проектов.
- Структура `entities/` для разных типов доменов (B2B клиенты vs роботы vs документы).
- Правила wikilinks и нормализации имён.
- Триггеры для curator-а (когда автоматически разрешать конфликт, когда звать архитектора).
- Метрики здоровья brain (как понять, что он начал деградировать).
- Миграция с CHANGELOG-only на CHANGELOG + brain (что происходит со старыми записями).
- Конфигурация двойного brain (Уровень A + Уровень B) — для продуктов-фабрик.
- Правила, при которых brain может быть приостановлен или закрыт.

Эти разделы **появятся в полном тексте паттерна** после того, как мы получим реальный опыт в КИБА (активная обкатка) и MARS (медленная обкатка).
