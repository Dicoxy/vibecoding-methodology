# Паттерн: Agent Context — контекстный файл для AI-оператора

## Проблема

AI-оператор (Claude Code, Cline, Cursor) начинает каждую сессию с чистого листа. У него нет памяти о проекте: какая архитектура, какие конвенции, что уже проверено и работает, что сейчас в фокусе, куда не лезть.

RAG частично решает проблему — агент может запросить контекст из базы знаний. Но RAG требует правильного запроса, а агент в начале сессии не знает, что спросить. Кроме того, RAG возвращает чанки, а не целостную картину проекта.

Результат без контекстного файла:
- Агент ломает то, что уже проверено и работает
- Нарушает конвенции (использует не ту библиотеку иконок, забывает dark mode)
- Лезет в файлы другого оператора
- Не понимает архитектуру и принимает локально "правильные", но системно вредные решения

## Когда применять

- В проекте есть AI-оператор, который реализует блоки
- Проект имеет архитектурные решения, конвенции и verified факты
- Несколько операторов работают параллельно (нужны границы)
- Проект на Tier 2+ (8+ блоков) — слишком много контекста для устного объяснения

Не нужен для:
- Одноразовых задач (скрипт, утилита)
- Проектов без AI-оператора (только архитектор)
- Tier 1 проектов (1–5 блоков) — контекст помещается в один контракт

## Решение

Статический файл `CLAUDE.md` (или `AGENTS.md`) в корне проекта, который AI-оператор читает в начале каждой сессии. Курируется вручную оркестратором или архитектором — не генерируется автоматически.

### Почему статический файл, а не RAG

| RAG-запрос | Статический файл |
|---|---|
| Агент должен знать что спросить | Агент читает всё сразу |
| Возвращает чанки, не целостную картину | Даёт полный контекст проекта |
| Может не найти нужное | Всё критичное — гарантированно |
| Задержка на поиск | Мгновенно |

Статический файл и RAG не конкурируют — они дополняют друг друга. Файл даёт базовый контекст (архитектура, конвенции, запреты). RAG даёт детали по запросу (конкретные решения, история изменений).

### Структура CLAUDE.md

#### 1. Проект (2–3 строки)

Что это, на чём написано, где живёт.

```markdown
## Проект
[Название] — [одна строка что делает].
Backend: [стек]. Frontend: [стек].
Сервер: [IP]. Репо: [URL].
```

Минимум слов, максимум конкретики. Агент за 5 секунд понимает контекст.

#### 2. Архитектура (ASCII-диаграмма + ключевые файлы)

Пайплайн данных или структура системы — одна диаграмма. Плюс список ключевых файлов с однострочными описаниями.

```markdown
## Архитектура

\```
Input → Processor → Validator → Storage → API → Frontend
\```

Ключевые файлы:
- `api/processor.js` — основная логика обработки
- `api/validator.js` — валидация данных
- `dashboard/src/services/api.js` — адаптер API для фронта
```

Не вся архитектура — только то, что агенту нужно для навигации.

#### 3. Verified факты (НЕ ЛОМАТЬ)

Священный список. Факты, подтверждённые вручную, которые агент не имеет права ломать. Каждый факт — одна строка с объяснением почему это важно.

```markdown
## Verified факты (НЕ ЛОМАТЬ)

- **PUDU HMAC:** path WITH query params. Менять = ломать auth.
- **React.lazy():** все страницы через Suspense — одна битая не роняет всё.
- **api.js адаптер:** владелец — Гриша. НЕ менять без согласования.
```

Формат: **что** + **почему нельзя трогать** (или что сломается). Если факт не объясняет почему — он бесполезен, агент проигнорирует.

→ Подробнее: паттерн [verified-facts.md](verified-facts.md) (TODO)

#### 4. Конвенции кода

Правила, которые агент должен соблюдать в каждом файле. Группировка по слоям (backend, frontend, общие).

```markdown
## Конвенции

### Backend
- `'use strict';` в начале каждого файла
- Модули stateless — не хранят состояние
- Тесты рядом: `module/__tests__/`

### Frontend
- ВСЁ на русском в UI
- Dark mode: всегда dark: варианты
- Иконки: @phosphor-icons/react — ТОЛЬКО эта библиотека
```

Конкретные правила, не философия. "Используй phosphor-icons" — конвенция. "Пиши чистый код" — не конвенция.

#### 5. Текущий контракт / фокус

Что агент должен делать прямо сейчас. Ссылка на контракт или краткое описание задачи.

```markdown
## Текущий контракт

Wave 10, block 33 — robots-page-situation
Спека: `docs/contracts/wave-10/33-robots-page-situation.md`
Статус: в работе
```

Обновляется при каждой смене контракта/волны/фазы.

#### 6. Запреты (глобальные)

Что агент не должен делать НИКОГДА, вне зависимости от контракта. Отличается от §4 контракта: §4 — запреты конкретного блока, здесь — запреты проекта.

```markdown
## Чего НЕ делать

- Не менять api.js без согласования с архитектором
- Не менять коннекторы (pudu.js, viggo.js)
- Не менять миграции которые уже применены
- Не коммуникировать с клиентами или вендорами
- Не использовать console.log в production коде
```

### Дополнительный файл: current-focus.md

Для проектов с частой сменой фокуса — отделяемый файл `.claude/current-focus.md`. Обновляется чаще чем CLAUDE.md: при каждой смене контракта, при завершении блока.

```markdown
# Текущий фокус

## Оператор — [имя/роль]
Текущий контракт: [ссылка] — [статус]
Следующий: [определяется / ссылка]

## Что готово
- [Список готовых компонентов/API]

## Что НЕ готово
- [Список незавершённого — чтобы агент не опирался на недоделанное]
```

Разделение: CLAUDE.md — стабильный контекст (архитектура, конвенции, verified). current-focus.md — оперативный контекст (что сейчас, что дальше). CLAUDE.md меняется раз в неделю. current-focus.md — ежедневно.

### Процесс обновления

**Кто обновляет:** Оркестратор или архитектор. Не агент — агент потребитель контекста, не его автор.

**Когда обновлять CLAUDE.md:**
- Новый verified факт → добавить в секцию 3
- Новая конвенция → добавить в секцию 4
- Архитектурное изменение → обновить секцию 2
- Новый глобальный запрет → добавить в секцию 6

**Когда обновлять current-focus.md:**
- Контракт завершён → обновить статус
- Новый контракт назначен → обновить ссылку
- Блок готов → переместить из "не готово" в "готово"

**Как обновлять:** Вручную. Осознанная курация — не автогенерация. Автогенерация из CHANGELOG загрязняет контекст нерелевантными деталями. Оркестратор решает, что агенту нужно знать.

### Размер и токены

CLAUDE.md должен помещаться в контекстное окно агента с запасом на контракт и код. Практическое ограничение:

| Агент | Контекст | Бюджет на CLAUDE.md |
|-------|----------|-------------------|
| Claude Code | 200K | до 5K токенов (~4 стр.) |
| Cline/Cursor | 128K | до 3K токенов (~2.5 стр.) |
| DeepSeek | 64K | до 1.5K токенов (~1 стр.) |

Если CLAUDE.md разрастается — выносить детали в отдельные файлы (CONTRACT_GUIDE.md, ARCHITECTURE.md), оставляя в CLAUDE.md только ссылки и критичные факты.

## Связь с методологией

Паттерн конкретизирует **блок 03 (Changelog + RAG)** методологии v0.5:

- **Блок 03, секция 3.5** описывает "промптовый шаблон" для агента как RAG-запрос. Agent Context показывает, что статический файл эффективнее RAG для базового контекста проекта.
- **Блок 03** фокусируется на автоматическом извлечении контекста (RAG). Agent Context — ручная курация. Оба подхода дополняют друг друга: файл для стабильного контекста, RAG для деталей по запросу.
- Связь с паттерном [knowledge-stack.md](knowledge-stack.md): CLAUDE.md — один из слоёв Knowledge Stack (`source_type: context`).
- Связь с паттерном [contract-spec.md](contract-spec.md): §4 контракта (запреты блока) дополняет секцию 6 CLAUDE.md (глобальные запреты проекта).

## Антипаттерны

### ❌ "Автогенерация из CHANGELOG"
CLAUDE.md, собранный автоматически из последних записей CHANGELOG — шум. Оркестратор фильтрует и курирует, иначе агент читает 50 строк и не выносит ничего полезного.

### ❌ "Вся архитектура в CLAUDE.md"
Полная документация архитектуры — в отдельном файле. В CLAUDE.md — пайплайн-диаграмма и ключевые файлы. Агенту нужна навигация, не учебник.

### ❌ "CLAUDE.md не обновляется"
Устаревший контекст хуже отсутствующего. Если verified факт добавлен, а CLAUDE.md не обновлён — агент не узнает о запрете и сломает. Обновление CLAUDE.md — часть ритуала смены контракта.

### ❌ "Конвенции как философия"
"Пиши чистый код" — не конвенция. "Используй @phosphor-icons/react, не lucide" — конвенция. Конкретные правила, которые можно проверить механически.

### ❌ "Агент обновляет свой контекст"
Агент — потребитель контекста. Если агент начинает редактировать CLAUDE.md, он может удалить запреты или verified факты, которые ему "мешают". Контекст курирует оркестратор.
