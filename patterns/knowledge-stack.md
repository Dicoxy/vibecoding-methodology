# Паттерн: Knowledge Stack — разделение проектного знания на слои

## Проблема

CHANGELOG растёт линейно с каждой сессией. К 300–500 строкам в нём смешиваются: технические диффы, архитектурные решения, продуктовые инсайты, клиентские факты, процессные уроки. Решения тонут в техлоге. RAG помогает искать, но garbage in → garbage out: если знания не разделены по типу, поиск зашумлён нерелевантными чанками.

Проблема обостряется на Tier 2+ (8+ блоков, 2+ участника): техлог обновляется ежедневно, решения — раз в неделю, продуктовые инсайты — нерегулярно. Разные типы знаний имеют разный срок жизни и разную аудиторию, но лежат в одной куче.

## Когда применять

- CHANGELOG превысил ~300–400 строк (Tier 2+)
- В проекте есть AI-оператор (Маша, Claude Code) — ему нужен отдельный контекст
- Есть клиентское/продуктовое знание, которое не должно тонуть в техлоге
- Архитектурные решения принимаются часто и их нужно находить за секунды

Для Tier 1 (1–5 блоков, один человек) паттерн избыточен — достаточно одного CHANGELOG с хорошей структурой.

## Решение

Разделить проектное знание на слои. Каждый слой — для своей аудитории, с разной частотой обновления, в отдельном файле или директории.

### Обязательные слои (минимум для Tier 2)

**1. CHANGELOG.md — техлог (только код)**

Что изменено, коммиты, verified, known issues. Только код и инфраструктура.

НЕ писать сюда: архитектурные решения (→ ADR), продуктовые инсайты (→ дневник), процессные уроки (→ workflow).

Аудитория: разработчики, AI-агенты. Срок жизни записей: недели.

**2. decisions/ — ADR (одно решение = один файл)**

Architecture Decision Records. Нумерация: `001-название.md`, `002-название.md`. Append-only: решение не редактируется, а заменяется новым ADR со статусом "superseded by NNN".

Формат:
```markdown
# ADR-NNN: Название решения

**Дата:** YYYY-MM-DD
**Статус:** Accepted | Superseded by NNN | Deprecated
**Автор:** кто принял

## Контекст
Почему возник вопрос, какие ограничения, какие альтернативы рассматривались.

## Решение
Что решили и почему именно это.

## Последствия
Что это меняет, какие риски, что придётся сделать.
```

Когда писать: при любом решении, которое влияет на архитектуру, стек, процесс или бизнес-модель. Правило: если решение объясняет "почему так" — это ADR, не запись в CHANGELOG.

Аудитория: оркестратор, архитектор, новые участники. Срок жизни: годы.

**3. CLAUDE.md / AGENTS.md — контекст для AI-оператора**

Живой документ, который AI-агент читает в начале каждой сессии. Содержит: архитектуру проекта, verified факты (НЕ ЛОМАТЬ), конвенции кода, текущий контракт/фокус, явные запреты.

Обновляется оркестратором при смене фазы, волны или контракта. Не генерируется автоматически — курируется вручную.

Аудитория: AI-оператор (Claude Code, Cline, etc). Срок жизни: постоянно обновляется.

→ Подробнее: паттерн [agent-context.md](agent-context.md) (TODO)

### Опциональные слои (по потребности)

**4. PRODUCT_DIARY.md — продуктовый дневник**

Инсайты от клиентов, изменения позиционирования, бизнес-решения. Хронологический, с категориями: `[КЛИЕНТ]`, `[ПРОДУКТ]`, `[БИЗНЕС]`, `[ВЕНДОР]`, `[ИНСАЙТ]`, `[ПРОЦЕСС]`.

Когда нужен: если есть клиенты, если продукт развивается, если бизнес-контекст влияет на техническую работу.

Аудитория: архитектор/основатель. Срок жизни: годы.

**5. CONTRACT_GUIDE.md — правила контрактов**

Как писать контракты для AI-оператора. Обязательные секции, антипаттерны, эталонный пример.

Когда нужен: если есть AI-оператор, которому пишутся контракты.

→ Подробнее: паттерн [contract-spec.md](contract-spec.md) (TODO)

**6. WORKFLOW.md — операционные правила**

Как вести CHANGELOG, как обновлять TASKS, как коммитить, как использовать RAG. Процессный документ.

Когда нужен: при 2+ участниках или при сложном процессе.

### Правило перенаправления

Ключевой механизм, предотвращающий свалку "всё в CHANGELOG":

```
Пишу решение в CHANGELOG → СТОП → это ADR → выношу в decisions/NNN-*.md
Пишу про клиента в CHANGELOG → СТОП → это дневник → выношу в PRODUCT_DIARY.md
Пишу про процесс в CHANGELOG → СТОП → обновить WORKFLOW или CONTRACT_GUIDE
```

Правило работает как триггер: если замечаешь что запись не про код — перенаправь. Лучше перенаправить позже, чем не перенаправить вовсе. Старые записи в CHANGELOG не трогать (append-only) — просто с этого момента писать по правилам.

### Автоматическая проверка (опционально)

Git pre-commit hook, который предупреждает:
- Решение в CHANGELOG без ADR → "Это архитектурное решение? Вынеси в decisions/"
- Контракт без §4/§5 → "Добавь секцию 'Что НЕ делать' и 'Тест приёмки'"

Не блокирует коммит — предупреждает. Люди игнорируют правила под давлением дедлайна, хук напоминает.

## Файловая структура

```
project/
├── docs/
│   ├── CHANGELOG.md              ← Техлог (только код)
│   ├── PRODUCT_DIARY.md          ← Продуктовый дневник (опц.)
│   ├── WORKFLOW.md               ← Операционные правила (опц.)
│   ├── CONTRACT_GUIDE.md         ← Правила контрактов (опц.)
│   ├── decisions/                ← ADR
│   │   ├── 001-название.md
│   │   ├── 002-название.md
│   │   └── ...
│   └── contracts/                ← Контракты для оператора
│       ├── wave-1/
│       └── ...
├── CLAUDE.md                     ← Контекст для AI-оператора
└── TASKS.md                      ← Текущие задачи
```

Конкретная структура адаптируется под проект. Принцип: каждый тип знания — в своём месте, навигация по типу, не по дате.

## RAG-индексация

Все слои индексируются в единую RAG-коллекцию с полем `source_type` в метаданных:

| Слой | source_type | Что ищем |
|------|-------------|----------|
| CHANGELOG | `implementation` | Что конкретно сделано, verified факты |
| decisions/ | `decisions` | Почему так решили, альтернативы |
| PRODUCT_DIARY | `product` | Клиентские инсайты, бизнес-контекст |
| WORKFLOW | `methodology` | Как работать, правила процесса |
| CLAUDE.md | `context` | Текущее состояние проекта |

Перед RAG-запросом определи категорию: "я ищу факт, решение, процесс или клиентский инсайт?" → фильтруй по `source_type` до семантического поиска.

→ Подробнее: блок 03 методологии, секция 3.7 "Document-aware routing"

## Связь с методологией

Паттерн развивает и конкретизирует **блок 03 (Changelog + RAG)** методологии v0.5:

- **Блок 03** описывает 6 абстрактных слоёв знаний. Knowledge Stack даёт конкретную файловую структуру и форматы.
- **Блок 03** предполагает что слои 5 (Known Issues) и 6 (Decisions) — производные от CHANGELOG через RAG-парсинг. Knowledge Stack выносит решения в отдельные файлы (ADR) — они становятся first-class citizens, а не производными.
- **Правило перенаправления** — новый операционный механизм, которого в блоке 03 нет.
- **Pre-commit hook** — автоматизация правил, которая в блоке 05 (Процесс) не описана.

## Антипаттерны

### ❌ "Один CHANGELOG на всё"
Смешивание решений, техлога, продуктовых инсайтов в одном файле. При 500+ строках поиск нужного — археология.

### ❌ "Перепишем CHANGELOG по-новому"
Не надо. Старые записи не трогать. С момента внедрения — писать по правилам перенаправления. Старые решения выносить в ADR по мере обнаружения (при обращении к ним).

### ❌ "ADR — бюрократия"
ADR — это 10–20 строк markdown. Время написания: 5 минут. Время поиска решения без ADR: 15–30 минут (grep по CHANGELOG + восстановление контекста). Окупается после второго обращения.

### ❌ "Сделаем все слои сразу"
Начни с CHANGELOG + decisions/. Остальное — по потребности. PRODUCT_DIARY не нужен если нет клиентов. WORKFLOW не нужен если работаешь один. Не создавай пустые файлы "на будущее".

## Происхождение

Паттерн выведен из MARS (апрель 2026): 10,800+ строк CHANGELOG, 7 ADR, 33 контракта, 1,500+ RAG-чанков. Кристаллизовался как ADR-007 когда 272 решения оказались закопаны в техлоге. Формализован в MARS METHODOLOGY.md и WORKFLOW.md §5.

Внешние влияния: ADR (Fowler, AWS, Backstage), Specification Engineering (эволюция vibe coding 2026), AGENTS.md / CLAUDE.md (context engineering для AI coding agents).
